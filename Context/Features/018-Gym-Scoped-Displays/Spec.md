# Feature 018: Gym-Scoped Displays

**Status:** Draft
**Created:** 2026-04-06
**Supersedes:** any prior 018 / "multiple-instance-display" exploration. Ignore those docs.
**Related:** F008 (Display Broadcast Infrastructure), F009 (Display Route + Idle Mode)

---

## Overview

Introduce a first-class **Gym** entity that represents a physical place where lifting happens. A user belongs to zero or more gyms; a display belongs to exactly one gym. The Supabase Realtime broadcast channel that powers the remote TV is rescoped from a single instance-wide `display` channel to per-gym channels named `display:gym:{gym_id}`. Users pick which gym they are training at when they start a workout (sticky default to last-used). Idle-mode schedules and "push to display" focus events flow only to that gym's channel.

This is the model fix for the friends-and-family deployment problem: today, every member of a Supabase instance is mixed onto one TV, so users who lift in their garage pollute the home-gym TV and members of the home gym never see each other on TVs at other locations. Gyms partition the broadcast graph along the boundary that already exists in the physical world.

## Problem Statement

The current display infrastructure (F008/F009) treats a Supabase instance as a single venue:

- One broadcast channel: `display`
- One opt-in flag: `user_profiles.display_visible`
- One TV URL: `/display`
- Idle-mode Edge Function queries today's scheduled sessions across **all** users of the instance, regardless of where they actually lift

For a single-location gym this is fine. For a friends-and-family instance with members spread across multiple physical locations (home gyms, commercial gyms, the host's basement), the model breaks in both directions:

1. **Pollution:** A member doing a garage workout shows up on the host's TV as if they were in the room.
2. **Exclusion:** A member who installs a TV at their own location has nowhere to point it -- they would just see the same single feed.

The PRD constraints from F008/F009 must continue to hold:

- **Constraint 1:** the display never writes to the database
- **Constraint 2:** the display never authenticates
- **Constraint 3:** snapshots are ephemeral (no persistence)

The fix needs to add the gym dimension without breaking those constraints. The display can still be a passive, unauthenticated subscriber as long as it knows _which gym channel to listen to_.

## Goals

1. Partition workout broadcasts by physical location, not by Supabase instance.
2. Let a single Supabase instance host arbitrarily many gyms without code or config changes per gym.
3. Keep the display passive, unauthenticated, and database-quiet.
4. Replace the global `display_visible` opt-out with a more expressive per-workout choice ("publish to gym X" / "private").
5. Lay schema foundations for future gym features (location-based auto-pick, gym-scoped programs, owner moderation) without building them yet.

## Non-Goals (this iteration)

- **Location inference** (geolocation lat/lng or wifi BSSID auto-pick). Resolved as `RD-14` below: deferred. Manual pick at workout start with sticky default is the v1 model.
- **Invitations / private gyms.** v1 is self-service: any authenticated user can create or join any gym on the instance. Token-based invitations are a future iteration.
- **Per-gym content sharing** (programs, templates, exercises scoped to a gym). Orthogonal to F016 Public Visibility.
- **Gym branding** (logos, colors, custom display themes). Future.
- **Cross-gym aggregations** (a "global" feed that shows every gym). Out of scope.
- **Per-gym roles or permissions** beyond owner/member. v1 has owner (created the gym) and member (joined it). No "coach", "admin", or "guest" tiers.
- **Mid-workout gym change.** Out of scope for v1 -- gym is locked at workout start. Users finish and start a new workout if they switch venues mid-session. Surfaced for future consideration.

## User Stories

1. **As a member of a friends-and-family instance**, I want to create a "Home Gym" and a "Garage" gym so that workouts I do at each location appear on the right TV.
2. **As a member**, I want to see a list of all gyms on my instance and join the ones I train at, so I do not need an admin to set me up.
3. **As a member starting a workout**, I want a quick picker that defaults to wherever I trained last, so 90% of the time I tap "Start" and it just works.
4. **As a member at a new venue**, I want to override the default and pick a different gym (or "Private") at workout start, so my session goes to the right place -- or nowhere at all.
5. **As a member with no gyms yet**, I want my workout to log normally without publishing anywhere, so I am never blocked from training because of gym setup.
6. **As a host running a TV**, I want to configure my TV with a single gym ID (in the URL) and have it only show sessions from members of that gym.
7. **As a host**, I want today's idle-mode schedule to show only the upcoming sessions for users currently checked into my gym, not every user on the instance.
8. **As a member**, I want a one-tap "Private (don't publish)" option in the gym picker so I can do a one-off workout that nobody sees, without leaving any gym.
9. **As a gym creator**, I want to be the owner of the gym I created so I can rename or delete it later.
10. **As any member**, I want to leave a gym I no longer train at, so I do not pollute its TV with workouts from elsewhere.

## Resolved Decisions
