# Feature 011: Deep Link Handler for `ardentforge://connect`

## Overview

Enable the `ardentforge://connect?url=...&key=...` deep link to function as a first-class onboarding path. When a user taps an invite link or scans a QR code that produces one, the app opens directly to the appropriate screen with credentials pre-populated and auto-validated. This completes the onboarding link chain: QR generation (010) produces links, this feature consumes them at the OS level.

## Problem Statement

The `ardentforge://connect` URL scheme is already generated by the QR code and invite link features (Refactor B / Feature 010), but tapping one of these links on a device with the app installed does nothing useful. The deep link plugin only registers the `auth` host, and the `onOpenUrl` listener only processes OAuth callbacks. Users must manually copy-paste invite links into the setup screen rather than having the app handle them natively.

## User Stories

1. **As an unconfigured user**, I tap an `ardentforge://connect` link shared by a friend, and the app opens to the setup screen with the server credentials already filled in and validating -- I just wait for the green check and sign in.

2. **As a configured user**, I tap an invite link for the same server I'm already on, and see a brief "Already connected to this server" confirmation -- no disruption to my session.

3. **As a configured user**, I tap an invite link for a different server, and the app navigates me to the backend settings where I can review and confirm the switch (with the standard data-wipe warning on Tauri).

4. **As a browser user**, I visit `/connect?url=...&key=...` and get the same behavior as the native deep link -- redirect to setup or settings with pre-populated values.

5. **As a user who receives a malformed link**, I see an "Invalid invite link" toast rather than a crash or silent failure.

## Requirements

### Must Have

- **M1:** Register `ardentforge://connect` as a handled deep link host in Tauri config and Android manifest
- **M2:** Extend the `onOpenUrl` listener to dispatch based on URL host/path (`auth` vs `connect`)
- **M3:** Implement onboarding handler that parses `url` and `key` query params from `ardentforge://connect` URLs
- **M4:** Unconfigured app: navigate to `/setup` with pre-populated Supabase URL and key, auto-trigger validation
- **M5:** Configured app, same instance: show "Already connected to this server" toast, no navigation
- **M6:** Configured app, different instance: navigate to backend settings with pre-populated change form
- **M7:** Malformed deep link (missing params): show "Invalid invite link" toast
- **M8:** Add TanStack Router `/connect` route for browser-mode equivalent
- **M9:** `/connect` route applies same logic: unconfigured -> setup, configured same -> toast, configured different -> settings
- **M10:** Setup screen accepts `url` and `key` via route search params for pre-population and auto-validation
- **M11:** Existing OAuth deep link flow (`ardentforge://auth/callback`) continues to work without regression

### Won't Have (this iteration)

- **W1:** Universal links / App Links (HTTPS-based deep links requiring domain verification) -- custom scheme only
- **W2:** Deep link for other actions beyond `connect` (e.g., `ardentforge://workout/123`)
- **W3:** Prompt to install the app if it's not installed (deferred to app store listing)
- **W4:** Native desktop app support -- this is a mobile + browser app only

## Testable Assertions

- **TA1:** `onOpenUrl` callback receiving `ardentforge://connect?url=X&key=Y` on an unconfigured app results in navigation to `/setup` with `url=X` and `key=Y` in search params
- **TA2:** `onOpenUrl` callback receiving `ardentforge://connect?url=X&key=Y` where X matches current config shows toast and does not navigate
- **TA3:** `onOpenUrl` callback receiving `ardentforge://connect?url=X&key=Y` where X differs from current config navigates to backend settings
- **TA4:** `onOpenUrl` callback receiving `ardentforge://connect` (missing params) shows "Invalid invite link" toast
- **TA5:** `onOpenUrl` callback receiving `ardentforge://auth/callback?code=Z` still triggers OAuth exchange (regression check)
- **TA6:** Browser navigation to `/connect?url=X&key=Y` on unconfigured app redirects to `/setup?url=X&key=Y`
- **TA7:** Browser navigation to `/connect?url=X&key=Y` on configured app (same) shows toast
- **TA8:** Browser navigation to `/connect?url=X&key=Y` on configured app (different) redirects to profile/settings
- **TA9:** Setup screen mounted with `url` and `key` search params expands advanced section, pre-populates fields, and auto-triggers validation
- **TA10:** `tauri.conf.json` deep-link plugin config includes `connect` host entry

## Open Questions

1. **Backend settings pre-population**: The backend settings component (`BackendSettings`) currently uses local state for the change form. Pre-populating from a deep link requires either route search params on `/profile` or a transient Zustand store. _Recommendation: use a lightweight Zustand transient store to avoid coupling settings layout to route params._

## Dependencies

- **Feature 010 (QR Code Gen + Scan):** Produces the `ardentforge://connect` URLs that this feature handles. The `buildInviteLink` and `parseInviteLink` utilities from `src/lib/invite-link.ts` are reused directly.
- **Existing deep-link plugin:** Already initialized in `lib.rs` and configured in `tauri.conf.json`. This feature extends the existing config rather than adding a new plugin.
