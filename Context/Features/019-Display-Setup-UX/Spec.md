# Feature 019: Display Setup UX

**Status:** Draft
**Created:** 2026-04-07
**Related:** F018 (Gym-Scoped Displays), F008 (Display Broadcast Infrastructure), F009 (Display Route + Idle Mode), F010 (QR Code Gen + Scan)

---

## Overview

Close the discoverability gap left by F018: today nothing in the app surfaces the `/display/gym/{uuid}` URL that owners and members need to configure a TV, and `/display` is a static "DISPLAY NOT CONFIGURED — ask the gym owner for the display URL" dead-end even when the caller is the gym owner.

This feature does two things:

1. **Surface the display URL inline** — every gym row in Profile → GYMS → My gyms gains a "Show display" action that reveals the full URL, a Copy button, and a QR code. Works for all members, not just owners.
2. **Turn `/display` into a smart dispatcher** for authenticated users. Based on the caller's gym membership count:
   - **0 gyms** → a setup UI offering two mutually-exclusive paths: paste/scan a gym URL, or start a silent personal display.
   - **1 gym** → immediate `Navigate replace` to `/display/gym/{that gym id}`.
   - **2+ gyms** → a full-page chooser listing every membership.

The dumb-TV route (`/display/gym/$gymId`) stays exactly as F018 shipped it. Unauthenticated visitors to `/display` still see the legacy "DISPLAY NOT CONFIGURED" fallback page, unchanged.

## Problem Statement

F018 shipped gym-scoped broadcast channels and a canonical display route at `/display/gym/$gymId`, but it never closed the loop from "I am a gym owner" to "I have the URL in my hand." Concretely:

- **No URL surface.** `GymManagementSection` in profile lists memberships with Leave and Delete actions but no display-URL affordance. Owners (and anyone else running a TV) have no in-app way to find the UUID-shaped URL. The only hint is the legacy `/display` page telling them to "ask the gym owner" — a useless instruction when the reader _is_ the gym owner.
- **`/display` is a dead-end.** Post-launch commit d942080 added a "Display" item to the desktop sidebar nav pointing at `/display`, which currently renders the static "DISPLAY NOT CONFIGURED" page. A member who taps that nav item sees a page telling them to ask someone else for a URL they could have assembled themselves if the app had told them how.
- **Single-gym friction.** A user who only belongs to one gym should never have to pick. Their "which display?" answer is determined. But today they are either stuck at the not-configured page or would be forced through a chooser that has exactly one row.
- **Zero-gym orphan state.** Rare but possible: a user leaves their last gym, or the default Home gym is deleted. They can still log workouts (F018 RD-10) but have no path to set up a personal TV — the only fix is to go to profile and create a gym manually, which feels like ceremony for a user who just wants to watch their own lifts on a second screen.

The F018 PRD constraints still hold and must not be weakened:

- **Constraint 1:** The `/display/gym/$gymId` route never writes to the database
- **Constraint 2:** The `/display/gym/$gymId` route never authenticates
- **Constraint 3:** Snapshots are ephemeral

Everything this feature adds lives _outside_ the dumb-TV route — in profile UI, in a new authenticated dispatcher at `/display`, and in a new mutation the setup UI calls. The TV itself continues to boot with the exact same assumptions it had in F018.

## Goals

1. Make the `/display/gym/{uuid}` URL trivially discoverable for every gym a user belongs to.
2. Make `/display` the smart entry point for authenticated users, branching on membership count without ceremony.
3. Give zero-gym users a zero-ceremony fallback to a personal display without forcing them through a gym-creation form.
4. Preserve the dumb-TV contract: `/display/gym/$gymId` keeps all F018 constraints.
5. Preserve the unauthenticated `/display` fallback page so existing operator bookmarks and the dumb-TV no-config state behave unchanged.
6. Keep the feature additive — no F018 primitives change. Publisher, subscriber, idle Edge Function, channel naming, RLS, and the auto-enroll trigger are all untouched.

## Non-Goals (this iteration)

- **Display preview embed.** No "what the TV looks like" mini-view. The setup UI links out to the real display.
- **Multi-display per gym management.** F018 W6 — still deferred. All TVs pointed at the same gym continue to show identical state.
- **Display access tokens / signed URLs.** Anyone who sees a display URL can view the TV. This matches the F018 security posture; tightening it is a separate feature.
- **Display connection telemetry.** Knowing which displays are currently subscribed to a gym is outside scope.
- **Per-display customization** (themes, layouts, clock format picker). Out of scope; clock format search param already exists on the display route for this.
- **Rename of gyms via the setup UI.** Personal-gym naming happens once at auto-create time; rename is already available via profile gym management.
- **Short URLs / slugs for display URLs.** F018 RD-17 keeps gym names non-unique and leaves slug as a future compat concern; this feature does not earn the slug design cost.
- **Share-sheet integration** (native iOS/Android share menu). The Copy button and QR code cover the realistic paths for v1.
- **Notifications when a TV connects or disconnects.** Out of scope.

## User Stories

1. **As a gym owner**, when I visit Profile → GYMS → My gyms, I want to tap a "Show display" action on my gym row and immediately see the URL to point a TV at, so I do not have to ask anyone or dig through the URL bar.
2. **As any member (not just the owner)**, I want the same "Show display" action on gyms I'm a member of, because "running the TV" is a physical-location role, not a governance role.
3. **As a gym owner**, I want a QR code rendered next to the URL so I can scan it from a TV's phone-companion app or a second phone without transcribing a UUID.
4. **As a member of exactly one gym**, when I tap the Display sidebar nav item, I want to land on my gym's display without an intermediate chooser, because there is nothing to disambiguate.
5. **As a member of two or more gyms**, when I visit `/display`, I want to see a simple full-page chooser listing my memberships so I can pick one with one tap.
6. **As a member of zero gyms**, when I visit `/display`, I want two clear options: "Use an existing display URL" (paste or scan) OR "Start a personal display just for me", without being forced to create a gym from scratch.
7. **As a zero-gym member picking "personal display"**, I want the app to handle the gym creation silently — I type nothing, name nothing, click Create nothing. When I land on my display, my workouts broadcast there automatically.
8. **As a zero-gym member who wants to join a TV someone else set up**, I want to paste a `/display/gym/{uuid}` URL (or scan its QR) and have the app just navigate me there.
9. **As a zero-gym Tauri-app user**, I want to tap "Scan QR" and point my phone at the QR rendered on the other device's profile page to skip the paste step entirely.
10. **As an unauthenticated visitor** (a dumb TV with no login, an operator with an expired session, or someone who hit the URL without context), I want the existing "DISPLAY NOT CONFIGURED — ask the gym owner for the display URL" fallback page to still work, because that is the sensible error message for a TV booting without config.

## Open Questions

These are implementation-level and belong in Tech.md. Kept here for reference.

1. **Should `parseDisplayUrlInput` also accept production-origin URLs from _other_ deployments** (e.g., a user pastes a URL from a staging deployment into a prod app)? Lean yes — it still resolves to a gym ID, and the target gym is valid on any origin that talks to the same Supabase instance. The dispatcher should still navigate the user to the current origin's `/display/gym/{id}`, not cross-origin. **Owner:** Tech.md D-something.
2. **Where does the parser module live?** `src/lib/display-url.ts` is the natural home, but `src/lib/gym-channel.ts` already exists for channel-name manipulation. Are these related enough to colocate? Lean no — channel naming is an infrastructure concern; URL parsing is a UI concern. **Owner:** Tech.md.
3. **Should the dispatcher be a new file `src/routes/display/index.tsx` (replacing the current minimal file) or split into `src/components/display/dispatcher.tsx` + a thin route wrapper?** Lean toward the split so the route file stays thin and the component is unit-testable in isolation. **Owner:** Tech.md.
4. **Should "Show display" collapse other open rows automatically (S1)**, or is simultaneous multi-row open acceptable? Non-load-bearing UX detail. **Owner:** Tech.md.
5. **Feature flag or direct ship?** Lean direct — this is a pure additive UX fix on top of F018 and the blast radius is confined to one component plus one route. **Owner:** Tech.md.

## Out-of-Band Risks

| Risk                                                                                                                                   | Likelihood | Impact                                                                           | Mitigation                                                                                                                                                                                                                          |
| -------------------------------------------------------------------------------------------------------------------------------------- | ---------- | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Dev user copies a `http://localhost:5173/display/gym/{id}` URL to a real TV that cannot reach their machine                            | Medium     | TV just fails to load; no data loss                                              | **M24** dev-origin warning caption                                                                                                                                                                                                  |
| Tauri clipboard API fails silently on some Android WebView versions                                                                    | Low        | Copy feels broken                                                                | **M4** logs with `[display-setup]` prefix and shows an error toast telling the user to long-press-select; **S9** documents the `execCommand` fallback as a backlog option if this bites                                             |
| QR code is too small to scan from a couch                                                                                              | Low        | User has to walk up to their phone                                               | **M5** 192×192 minimum; revisit if users complain; tap-to-zoom is a cheap follow-up                                                                                                                                                 |
| User pastes a URL from a different Supabase instance into Panel A                                                                      | Low        | The gym UUID won't exist on their instance; TV shows "invalid gym id" on arrival | Acceptable — the dumb-TV `/display/gym/$gymId` route already handles invalid UUIDs via its own error UI                                                                                                                             |
| A 2+-gym user creates many personal displays over time, polluting the Browse all gyms list                                             | Low        | UX noise                                                                         | Per F018 RD-19, Browse all gyms is unscaled v1 with a pagination comment. If personal displays become a real source of noise, add an `is_personal` flag later (backlog). No v1 action.                                              |
| The dispatcher's 1-gym redirect fires during a stale `useGyms` cache state after the user leaves their last gym on another tab         | Low        | Brief mis-redirect; user lands on a display they can no longer publish to        | Acceptable — they'll see a valid display (they were a member recently) and can navigate back to setup via sidebar. TanStack Query's invalidation handles this on the next tick.                                                     |
| A malicious paste injects a JavaScript-URL into Panel A                                                                                | Low        | XSS                                                                              | Parser extracts only the UUID via Zod; navigation uses TanStack Router `params` (not raw string interpolation). No HTML is rendered from the input.                                                                                 |
| Personal-display auto-name collides with an existing gym name another user created                                                     | Low        | Both gyms coexist in Browse all gyms with identical labels                       | F018 RD-17 explicitly allows duplicate names; this is fine                                                                                                                                                                          |
| User with a very long `displayName` (e.g., 60 chars) triggers a personal-gym name longer than the 60-char `gyms.name` check constraint | Medium     | `createGym` fails with an RLS/validation error                                   | Name derivation helper clamps `displayName` to 48 chars before suffixing ` 's Training` (48 + 11 = 59, fits the 60-char limit). Fallback to `"My Training"` if the clamped result is empty. Unit test TA14 covers the clamp branch. |
