12 findings across 2 categories (8 [FIX], 4 [TASK]). One critical data bug (exercises lose

## Findings

### Fix-Now

#### [FIX] P17-001: `supports_1rm` camelization produces wrong key — exercises lose `supports1RM`
- **File:** `src/lib/adapter-utils.ts:18`, `src/lib/supabase-adapter.ts:131`
- **Severity:** Critical
- **Detail:** `snakeToCamel` regex `/_([a-z])/g` only matches underscores before lowercase
  letters, not digits. `supports_1rm` → `supports1Rm` (lowercase `m`). The domain type field
  is `supports1RM`. At `supabase-adapter.ts:131`, `r.supports1Rm as boolean` evaluates to
  `undefined` cast silently to boolean. Every exercise returned by `getExercises`,
  `getExercise`, and `createExercise` via SupabaseAdapter has `supports1RM: undefined`.
  The 1RM tracking tab shows no exercises; set scheme gating is broken. Tauri adapter is
  unaffected (reads `r.supports_1rm` from the typed struct directly, bypassing camelizeKeys).
  **Fix:** Read `raw['supports_1rm']` directly in `mapExercise`, or fix regex to
  `/_([a-z0-9])/g` AND update the read key to `r.supports1Rm` to match what the regex
  actually produces.
- **Status:** ✅ Fixed
- **Resolution:** `mapExercise` now reads `raw['supports_1rm']` directly, bypassing camelizeKeys for this digit-adjacent column.

#### [FIX] P17-002: `parseJsonOrValue` throws bare `SyntaxError` with no column context
- **File:** `src/lib/adapter-utils.ts:37-38`
- **Severity:** High
- **Detail:** `parseJsonOrValue` calls `JSON.parse(value)` without a try/catch. Malformed
  JSONB in any of 9 call sites across both adapters produces an unattributed
  `SyntaxError: Unexpected token X at position Y` — no column name, no entity type, no raw
  value. The Tauri adapter already has a local `parseJson(value, column)` helper that catches
  and rethrows with context. The new shared utility regresses this for the 4 Tauri call sites
  that switched to it (`toSessionTemplate:897`, `toActivityGroupFlat:936,940`, `toActivity:969`).
  **Fix:** Add `column: string` param and wrap with try/catch that rethrows with column name
  and raw value preview. All 9 call sites must pass the column name.
- **Status:** ✅ Fixed
- **Resolution:** `parseJsonOrValue` now requires `column: string` param and wraps `JSON.parse` in try/catch with column name + raw value preview. All 6 Supabase call sites updated. The 4 Tauri call sites were reverted to the local `parseJson` helper (see P17-006).

#### [FIX] P17-003: `intToBool` null coercion triggers without `console.warn` (acknowledged TODO)
- **File:** `src/lib/tauri-adapter.ts:607-611`
- **Severity:** Medium
- **Detail:** Project rule (error-handling.md §Adapter Boundary Fallbacks) requires
  `console.warn` at warn level when a safety-net coercion triggers for fields expected to be
  non-nullable. `intToBool` silently returns `false` for null inputs, including for `completed`
  (a field the inline comment says "should always provide a value" from Rust). A null `completed`
  makes in-progress workouts appear to have zero completions. The TODO `P14-021` acknowledges
  this but the refactor was the opportunity to fix it.
  **Fix:** Add `field: string` param; log `console.warn('[tauri-adapter] ${field}: expected
  int (0/1), got null -- using fallback ${fallback}')` before returning fallback.
- **Status:** ✅ Fixed
- **Resolution:** `intToBool` now requires `field: string` param and logs `console.warn('[tauri-adapter] ...)` when null fallback triggers. TODO comment removed. All 10 call sites updated with field names.

#### [FIX] P17-004: `mapGym`/`mapGymMember` pre-check `console.warn` implies tolerance but throws
- **File:** `src/lib/supabase-adapter.ts:259-273`, `src/lib/supabase-adapter.ts:278-292`
- **Severity:** Medium
- **Detail:** `if (r.id == null || ...) console.warn('[supabase-adapter] mapGym: missing required
  fields', raw)` fires, then falls through to `gymSchema.parse()` which throws, caught and
  re-logged as `console.error`. Two log lines for one root cause; the warn implies the code is
  tolerating the missing field, but it is not. Developers debugging a production Sentry trace
  have misleading signal.
  **Fix:** Remove the pre-check warn. The catch block already logs `console.error` with `raw`.
  Or: throw directly in the pre-check (no warn), skipping the Zod parse.
- **Status:** ✅ Fixed
- **Resolution:** Removed pre-check `console.warn` blocks from both `mapGym` and `mapGymMember`. The catch block's `console.error` is the sole log signal on failure.

#### [FIX] P17-005: 15 supabase-adapter mapper methods lack try/catch around Zod parse
- **File:** `src/lib/supabase-adapter.ts:119-507`
- **Severity:** High
- **Detail:** `mapExercise`, `mapWorkoutLog`, `mapLoggedActivityGroup`, `mapLoggedActivity`,
  `mapLoggedSet`, `mapUserProfile`, `mapSessionTemplate`, `mapActivityGroupFlat`, `mapActivity`,
  `mapOneRepMaxHistory`, `mapBlockWeek`, `mapProgramActivation`, `mapWeekStatus`, `mapShareLink`,
  `mapEventItem` all call Zod `parse()` without try/catch. A `ZodError` propagates with only
  Zod's internal field path — no entity type, no entity ID, no raw row. `mapProgram`,
  `mapBlock`, and `mapScheduledSession` do have try/catch (though without `console.error` before
  rethrow). Inconsistent coverage means malformed DB rows are untraceable in production.
  **Fix:** Wrap each mapper in try/catch that logs `console.error('[supabase-adapter] Failed to
  map <entity>:', err, raw)` and rethrows with entity type + ID context. Match the pattern
  already used by `mapProgram`.
- **Status:** ✅ Fixed
- **Resolution:** All 15 methods now wrapped in try/catch with `console.error('[supabase-adapter] Failed to map <entity>:', err, raw)` + rethrow. `mapProgram` and `mapBlock` also updated to add `console.error` to their existing catch blocks.

#### [FIX] P17-006: tauri-adapter `toSessionTemplate`/`toActivityGroupFlat`/`toActivity` use `parseJsonOrValue` instead of local `parseJson`
- **File:** `src/lib/tauri-adapter.ts:897`, `src/lib/tauri-adapter.ts:936`, `src/lib/tauri-adapter.ts:940`, `src/lib/tauri-adapter.ts:969`
- **Severity:** Medium
- **Detail:** The tauri-adapter defines a local `parseJson(value, column)` helper (line 558)
  specifically to produce context-rich errors. These four call sites import `parseJsonOrValue`
  from `adapter-utils` instead, bypassing column-name context. All other Tauri mapper functions
  that parse JSON columns use the local `parseJson` correctly. This inconsistency means these
  four columns (`rest_between_groups`, `time_cap`, `rest_between_rounds`, `rest_between_activities`,
  `set_scheme`) produce bare `SyntaxError` on bad data.
  **Fix:** Replace `parseJsonOrValue(r.x)` with `parseJson(r.x, 'x')` at all four call sites.
  If P17-002 is fixed to add a `column` param, `parseJsonOrValue` and `parseJson` can unify.
- **Status:** ✅ Fixed
- **Resolution:** All 4 call sites replaced with local `parseJson(r.x, 'x')`. The `parseJsonOrValue` import removed from tauri-adapter.ts.

### Missing Tasks

#### [TASK] P17-007: Add unit tests for `camelizeKeys` and `parseJsonOrValue` in `adapter-utils.test.ts`
- **File:** `src/lib/__tests__/` (new file: `adapter-utils.test.ts`)
- **Severity:** High
- **Detail:** Both utilities in `adapter-utils.ts` have zero unit tests. `camelizeKeys` is the
  foundational transform used by every Supabase adapter mapper. Key behaviors that must be
  verified: already-camelCase passthrough, null/undefined values, keys adjacent to digits
  (the `supports_1rm` case specifically), multiple consecutive underscores, leading underscores.
  `parseJsonOrValue` must verify: string path (`JSON.parse`), object-passthrough, and error
  behavior on malformed JSON. These are the two highest-value tests to add given the surface
  area they protect.
- **Status:** 📋 Task created
- **Resolution:** Added to `Context/Backlog/Ideas.md` as P17-007.

#### [TASK] P17-008: Add `mapScheduledSession` overrides malformed-JSON fallback test
- **File:** `src/lib/__tests__/supabase-adapter.test.ts`
- **Severity:** Medium
- **Detail:** `mapScheduledSession` (supabase-adapter.ts:417) wraps the `overrides` column parse
  in an inner try/catch that degrades to `undefined` + `console.warn`. The deleted
  `data-mapper.test.ts` (lines 2082-2110) verified this fallback explicitly with two tests:
  one for malformed JSON string, one for schema-invalid JSON. No remaining test exercises the
  catch branch. If the fallback is removed or the warn is dropped, it is undetectable.
  **Fix:** Add a `scheduledSessionRow` fixture variant with malformed JSON in `overrides`, assert
  `result.overrides === undefined` and that `console.warn` was called.
- **Status:** 📋 Task created
- **Resolution:** Added to `Context/Backlog/Ideas.md` as P17-008.

#### [TASK] P17-009: Fix Supabase adapter JSONB test fixtures to pass pre-parsed objects
- **File:** `src/lib/__tests__/supabase-adapter.test.ts`
- **Severity:** Medium
- **Detail:** `sessionTemplateRow` (line 166) and `templateActivityRow` (line 193) pass
  `JSON.stringify({...})` for JSONB columns (`rest_between_groups`, `time_cap`, `set_scheme`).
  PostgREST returns these columns as already-parsed objects, not strings. The existing tests
  exercise the string branch of `parseJsonOrValue` — the wrong code path for Supabase.
  If `parseJsonOrValue`'s object-passthrough branch ever broke, no Supabase adapter test
  would catch it.
  **Fix:** Update fixtures to pass pre-parsed objects: `rest_between_groups: { seconds: 120 }`
  rather than `JSON.stringify({ seconds: 120 })`. Add a second variant with strings to retain
  coverage of the Tauri string path.
- **Status:** 📋 Task created
- **Resolution:** Added to `Context/Backlog/Ideas.md` as P17-009.

#### [TASK] P17-010: Assert `supports1RM` in `getExercises` Supabase adapter happy-path test
- **File:** `src/lib/__tests__/supabase-adapter.test.ts:301`
- **Severity:** Medium
- **Detail:** The `getExercises` happy-path test asserts on `id`, `name`, `category`,
  `movementPattern`, and `muscleGroups` but not on `supports1RM`, `isBilateral`, `isCustom`,
  or `isPublic`. The `supports_1rm` camelization issue (P17-001) would have been caught here
  had this assertion existed. The Tauri adapter test does assert `result[0].supports1RM === true`
  (line 588), providing protection for the Tauri path but not the Supabase path.
  **Fix:** Add `supports1RM: true` (or false) to the exercise fixture and assert it in the
  `getExercises` result.
- **Status:** 📋 Task created
- **Resolution:** Added to `Context/Backlog/Ideas.md` as P17-010.

#### [FIX] P17-011: `console.warn` calls in tauri-adapter missing bracketed `[module-name]` prefix
- **File:** `src/lib/tauri-adapter.ts` (multiple warn call sites)
- **Severity:** Low
- **Detail:** Multiple `console.warn` calls in tauri-adapter use the form `'tauri-adapter: ...'`
  without the bracketed prefix `[tauri-adapter]` that the project error-handling convention
  requires. The `parseNoteTags` and `invokeCommand` catch blocks use the correct format; the
  null-timestamp warn calls in the to-domain mapper functions do not. Makes production log traces
  harder to grep.
  **Fix:** Standardize all warn calls to `console.warn('[tauri-adapter] ...')`.
- **Status:** ✅ Fixed
- **Resolution:** All 16 `console.warn('tauri-adapter: ...)` calls updated to `console.warn('[tauri-adapter] ...')` bracket format.

#### [FIX] P17-012: `(err as Error).message` unsafe cast in `mapGym`/`mapGymMember` catch blocks
- **File:** `src/lib/supabase-adapter.ts:272`, `src/lib/supabase-adapter.ts:290`
- **Severity:** Low
- **Detail:** `throw new Error(\`...: ${(err as Error).message}\`)` — if `err` is not an `Error`
  instance (e.g. a plain string or a `PostgrestError`), `.message` is `undefined`, producing
  `"Failed to map gym row (id=abc): undefined"`. Other mappers in the same file already use the
  guarded form `err instanceof Error ? err.message : String(err)` correctly.
  **Fix:** Replace `(err as Error).message` with `err instanceof Error ? err.message : String(err)`
  at both catch sites.
- **Status:** ✅ Fixed
- **Resolution:** Both catch blocks in `mapGym` and `mapGymMember` updated to use guarded `err instanceof Error ? err.message : String(err)`.
