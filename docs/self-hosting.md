# Self-Hosting Ardent Forge

Ardent Forge is designed for self-hosting. You can run the full stack on your own infrastructure using Docker Compose, or bring your own Supabase Cloud project paired with any static hosting provider. Both paths produce a fully functional instance that any Ardent Forge mobile or web client can connect to.

## Path 1: Docker Compose (Recommended)

### Prerequisites

- Docker Engine 24+ and Docker Compose v2+
- A server or machine with ports 80 and 443 open
- A domain name pointed at your server (for automatic TLS; `localhost` works for local testing)
- `openssl` installed (for key generation)

### Quick Start

1. Clone the repository:

   ```bash
   git clone https://github.com/rghsoftware/ardent-forge.git
   cd ardent-forge
   ```

2. Copy the example environment file:

   ```bash
   cp .env.example .env
   ```

3. Generate secrets (Postgres password, JWT secret, anon key, and service-role key):

   ```bash
   bash scripts/generate-keys.sh
   ```

   This script generates a random 32-byte Postgres password, a 48-byte JWT signing secret, and two Supabase JWTs (anon and service_role) derived from that secret. All four values are written directly into your `.env` file.

4. Edit `.env` and set `SITE_URL` to your public domain (e.g., `https://forge.example.com`). Review any other values you want to customize (SMTP, Google OAuth, etc.).

5. Start the stack:

   ```bash
   docker compose up -d
   ```

6. Verify the deployment:
   - Open `SITE_URL` in your browser. You should see the Ardent Forge web app.
   - Supabase Studio is available at `SITE_URL/studio` (if `STUDIO_ENABLED=true`).
   - Register a new account to confirm auth is working.

### Container Architecture

The Docker Compose stack runs 9 containers:

| Container    | Image                                              | Role                                               |
| ------------ | -------------------------------------------------- | -------------------------------------------------- |
| `db`         | `supabase/postgres:15.8.1.060`                     | PostgreSQL database with persistent named volume   |
| `kong`       | `kong:2.8.1`                                       | API gateway routing requests to Supabase services  |
| `auth`       | `supabase/gotrue:v2.175.0`                         | Authentication service (email, OAuth)              |
| `rest`       | `postgrest/postgrest:v12.2.8`                      | RESTful API over PostgreSQL                        |
| `realtime`   | `supabase/realtime:v2.34.48`                       | WebSocket subscriptions for live updates           |
| `studio`     | `supabase/studio:20250317-6a9ad62`                 | Supabase admin dashboard (optional)                |
| `migrations` | Custom (built from `docker/migrations/Dockerfile`) | Init container that applies SQL schema, then exits |
| `web`        | Custom (built from `docker/web/Dockerfile`)        | nginx serving the Vite production build            |
| `caddy`      | `caddy:2-alpine`                                   | Reverse proxy with automatic TLS termination       |

### Configuration Reference

All configuration lives in a single `.env` file. Variables are grouped below.

#### Required

| Variable            | Description                                                                                                      | Default                 |
| ------------------- | ---------------------------------------------------------------------------------------------------------------- | ----------------------- |
| `SITE_URL`          | Public URL for the deployment (used for TLS, OAuth redirects, cookie domain)                                     | `http://localhost:3000` |
| `POSTGRES_PASSWORD` | Database password. Generated by `generate-keys.sh`.                                                              | `changeme`              |
| `JWT_SECRET`        | Supabase JWT signing secret for minting and verifying auth tokens. Generated by `generate-keys.sh`.              | (placeholder)           |
| `ANON_KEY`          | Supabase publishable (anon) key, safe to expose to browsers and mobile clients. Generated by `generate-keys.sh`. | (placeholder)           |
| `SERVICE_ROLE_KEY`  | Supabase service-role key with full database access. Never expose to clients. Generated by `generate-keys.sh`.   | (placeholder)           |

#### Postgres

| Variable        | Description                             | Default    |
| --------------- | --------------------------------------- | ---------- |
| `POSTGRES_HOST` | Hostname of the Postgres container      | `db`       |
| `POSTGRES_PORT` | Postgres port inside the Docker network | `5432`     |
| `POSTGRES_DB`   | Database name                           | `postgres` |

#### Optional: Supabase Studio

| Variable         | Description                                | Default |
| ---------------- | ------------------------------------------ | ------- |
| `STUDIO_ENABLED` | Enable the Supabase Studio admin dashboard | `true`  |

#### Optional: SMTP

Required for email-based auth (magic links, email confirmations). Without SMTP, set `GOTRUE_MAILER_AUTOCONFIRM` to `true` in the compose file to skip email verification.

| Variable            | Description          | Default |
| ------------------- | -------------------- | ------- |
| `SMTP_HOST`         | SMTP server hostname | (unset) |
| `SMTP_PORT`         | SMTP server port     | `587`   |
| `SMTP_USER`         | SMTP username        | (unset) |
| `SMTP_PASS`         | SMTP password        | (unset) |
| `SMTP_SENDER_EMAIL` | Sender email address | (unset) |

#### Optional: Google OAuth

| Variable               | Description                | Default |
| ---------------------- | -------------------------- | ------- |
| `GOOGLE_CLIENT_ID`     | Google OAuth client ID     | (unset) |
| `GOOGLE_CLIENT_SECRET` | Google OAuth client secret | (unset) |

When `GOOGLE_CLIENT_ID` is set, the auth container automatically enables Google sign-in. The OAuth redirect URI is `${SITE_URL}/auth/v1/callback`.

### Updating

Pull the latest changes and rebuild:

```bash
git pull
docker compose up -d --build
```

The migration runner init container re-runs on every `docker compose up`. It detects previously applied migrations and only applies new ones, so updates that include schema changes are handled automatically.

### Troubleshooting

**Port 80/443 already in use**

Another service (Apache, nginx, or another reverse proxy) is binding those ports. Stop the conflicting service, or adjust the Caddy port mappings in `docker-compose.yml`.

**DNS not resolving or TLS certificate errors**

Caddy provisions TLS certificates automatically via Let's Encrypt. For this to work, your domain's DNS A record must point to your server's public IP, and ports 80/443 must be reachable from the internet. For local testing, use `http://localhost:3000` as your `SITE_URL` instead.

**Database migration failures**

Check the migration runner logs:

```bash
docker compose logs migrations
```

Common causes: incorrect `POSTGRES_PASSWORD` in `.env`, or the database container not yet healthy when the runner starts (the compose file uses health checks, but network issues can interfere).

**Viewing container logs**

```bash
docker compose logs -f            # all containers
docker compose logs -f auth       # specific service
docker compose logs -f kong rest  # multiple services
```

**Resetting all data**

> **WARNING**: This destroys the database and all stored data. Your data cannot be recovered.

```bash
docker compose down -v
```

Then re-run `docker compose up -d` for a clean start.

## Path 2: Supabase Cloud + Static Hosting

### Prerequisites

- A Supabase account at [supabase.com](https://supabase.com)
- A static hosting provider (Vercel, Netlify, Cloudflare Pages, or similar)
- Supabase CLI (via `bunx supabase`)

### Steps

1. Create a new Supabase project at [supabase.com](https://supabase.com). Note your project reference ID (visible in the project URL).

2. Clone the repository:

   ```bash
   git clone https://github.com/rghsoftware/ardent-forge.git
   cd ardent-forge
   ```

3. Link the CLI to your project:

   ```bash
   bunx supabase link --project-ref <your-project-ref>
   ```

4. Push the database schema:

   ```bash
   bunx supabase db push
   ```

5. Deploy the web app to your hosting provider with these settings:

   | Setting                         | Value                                                             |
   | ------------------------------- | ----------------------------------------------------------------- |
   | Build command                   | `bun run build`                                                   |
   | Output directory                | `dist`                                                            |
   | `VITE_SUPABASE_URL`             | `https://<your-project-ref>.supabase.co`                          |
   | `VITE_SUPABASE_PUBLISHABLE_KEY` | Your project's anon key (from Supabase dashboard, Settings > API) |

6. In the Supabase dashboard, go to Authentication > URL Configuration and add your deployed web app URL to the list of allowed redirect URLs.

### Enabling Google OAuth

**Docker path**: Set `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` in your `.env` file. The auth container picks these up automatically and configures the redirect URI as `${SITE_URL}/auth/v1/callback`. In the Google Cloud Console, add that same URL as an authorized redirect URI for your OAuth client.

**Supabase Cloud path**: In the Supabase dashboard, go to Authentication > Providers > Google. Enable it and enter your Google client ID and client secret. Add `https://<your-project-ref>.supabase.co/auth/v1/callback` as an authorized redirect URI in the Google Cloud Console.

## Mobile App Configuration

After deploying with either path:

1. Open the Ardent Forge app.
2. Go to Settings > Backend.
3. Enter your Supabase URL and publishable (anon) key:
   - **Docker path**: The Supabase URL is your `SITE_URL`. The anon key is the `ANON_KEY` value from your `.env` file.
   - **Supabase Cloud path**: Both values are in the Supabase dashboard under Settings > API.
4. Save. The app validates the connection and then connects to your self-hosted instance.

## Architecture Overview

The Docker path runs a self-contained Supabase stack behind a Caddy reverse proxy:

1. **Caddy** terminates TLS (with automatic Let's Encrypt certificates) and routes traffic. API requests go to Kong; web app requests go to the nginx container.
2. **Kong** acts as the API gateway for all Supabase services, routing to GoTrue (auth), PostgREST (REST API), and Realtime (WebSocket) based on the request path.
3. **GoTrue** handles authentication (email, OAuth, JWT issuance).
4. **PostgREST** exposes the PostgreSQL database as a RESTful API with row-level security enforced per JWT claims.
5. **Realtime** provides WebSocket connections for live database change subscriptions.
6. **PostgreSQL** stores all application data. A named volume (`pgdata`) persists data across container restarts.
7. **Migration runner** is an init container that applies SQL migrations from the repository on startup, then exits. It uses the `SERVICE_ROLE_KEY` (which never leaves the Docker network) for DDL permissions.
8. **Web app** is a multi-stage Docker build: the first stage runs `bun install && bun run build`, the second copies the static output into nginx. No Node.js runtime in production.
9. **Supabase Studio** (optional) provides a web-based admin dashboard for inspecting and managing the database.

## Remote Display

The `/display` route provides a public, no-authentication TV display for gym environments. It shows a clock, today's scheduled sessions, and active workout progress.

### Accessing the Display

Navigate to `https://<your-domain>/display` on any browser (smart TV, tablet, etc.). No login is required.

- **Default clock format:** 24-hour. Override with `?clock=12h` for 12-hour format.
- **Example:** `https://your-app.example.com/display?clock=12h`

### Edge Function: display-idle-snapshot

The `display-idle-snapshot` Edge Function broadcasts scheduled session data to the display every 60 seconds. Without it, the display shows only a clock (no schedule information).

The cron schedule is pre-configured in `supabase/config.toml`:

```toml
[functions.display-idle-snapshot]
verify_jwt = false

[functions.display-idle-snapshot.cron]
schedule = "*/1 * * * *"
```

Deploy the function with:

```bash
bunx supabase functions deploy display-idle-snapshot
```

### Requirements

- The Edge Function uses the `service_role` key (configured automatically via Supabase environment variables).
- The `display-idle-snapshot` function must be deployed for schedule data to appear on the display.
- No additional database migrations are required beyond the standard schema.
