## What to test

### camelizeKeys
1. Standard snake-to-camel: `{ created_at: 'x' }` → `{ createdAt: 'x' }`
2. Already-camelCase passthrough: `{ createdAt: 'x' }` → key unchanged
3. Multiple underscores: `{ foo_bar_baz: 1 }` → `{ fooBarBaz: 1 }`
4. Leading underscore: `{ _private: 'x' }` — assert the actual output (the regex won't match `_p` at start... wait, actually `_([a-z])` WOULD match `_p` in `_private`). Check the actual behavior.
5. **Digit-adjacent regression** (important): `{ supports_1rm: true }` — the regex `/_([a-z])/g` does NOT match `_1` because `1` is not `[a-z]`. So the key stays as `supports_1rm`. Assert this actual (broken) behavior and add a comment: `// known limitation: underscores adjacent to digits are not transformed`
6. Object value pass-through: values are not deep-transformed — `{ foo_bar: { nested_key: 1 } }` → `{ fooBar: { nested_key: 1 } }` (nested keys untouched)
7. Empty object: `{}` → `{}`

### parseJsonOrValue
1. String JSON input parses correctly: `parseJsonOrValue('{"a":1}', 'col')` → `{ a: 1 }`
2. Object passthrough: passing a pre-parsed object returns it unchanged (same reference or deep equal)
3. Array input: passing `[1, 2, 3]` returns it unchanged (it's not a string)
4. Malformed JSON throws: `parseJsonOrValue('{bad json}', 'overrides')` throws an error whose message contains `"overrides"` (the column name)
5. Malformed JSON error also contains the raw value in the message
